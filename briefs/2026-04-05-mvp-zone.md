# Design Brief: MVP Zone — Member Landing Page
**Date:** 2026-04-05
**Author:** Muse
**Route:** `kingofalldata.com/mvp-zone`
**Audience:** Authenticated Insiders (Level 1–4 GitHub Sponsors)

---

## Scope and Relationship to Prior Work

`2026-04-04-mvp-zone-design.md` specs the **namespace directory** — a public-facing view of who's in the ecosystem. This brief specs the **member landing page** — what a ring member sees when they arrive at `/mvp-zone` after sponsoring. These are two surfaces that can share a route (`/mvp-zone`) with different content rendered per authentication state:

- **Authenticated member:** member landing page (this brief)
- **Unauthenticated visitor:** public namespace directory (prior brief)

If a single route is too complex at this stage, `/mvp-zone` is public; `/mvp-zone/home` is member-only. Prefer the single route with auth-gated content.

---

## What This Page Is

The peer ring homepage. Not a dashboard. Not a SaaS management panel.

A sponsor clicked through, they're already in. This page answers: **what does being in actually mean, and where do I go next?** It should feel like arriving at the right place — calm, specific, unhurried. Like walking into a room where the other people already know what they're doing and you're welcome.

The emotional register: **a clubhouse with a server rack in the corner.** Warm and purposeful, not polished and hollow.

---

## Authentication Model

GitHub Sponsors OAuth. The user signs in via GitHub. The page reads their sponsorship tier from the GitHub Sponsors API, then renders tier-appropriate content.

- Level 0 (not sponsor, but visiting): show the public directory + a soft "you're not in yet" moment — a single CTA block at top, not a sales wall
- Level 1–4: full member view, tier badge visible, tier-locked content revealed progressively

No third-party tracking pixels. No analytics beacons. No avatar fetched from Gravatar or similar — only what comes from the GitHub API the user already authorized.

## Section 2: The Digest

**What it is:** Juno's weekly ops field report. Not a newsletter. Not marketing. A frank account of what the entity team did, what's blocked, what shipped.

**Delivery mechanism:** The digest lives on this page first, then optionally in the Keybase channel.

**Why not email:** Email requires an email address, an ESP, unsubscribe compliance, and deliverability infrastructure none of which exists yet. The digest is a Markdown file committed to a private repo (or a private GitHub Gist), then fetched and rendered here for authenticated members. No email required.

**Implementation approach:**
1. Juno commits the weekly digest as `digests/YYYY-WW.md` in a private `koad/insiders` repo (or as a secret GitHub Gist accessible via API)
2. On page load for authenticated members, the page fetches the latest digest via GitHub API using the member's token
3. Rendered as styled Markdown — the same document Juno actually committed
4. Members read it here. Optionally, Juno cross-posts to the Keybase channel as a link

**UX layout:**

```
DIGEST

Week 6 · 2026-04-05
────────────────────────────────────────────────────

What shipped this week:
  · Alice Phase 2A live on kingofalldata.com (Vulcan, 7d95c39)
  · Blog infrastructure PR open (#1 on kingofalldata-dot-com)
  · Day 6 content: Trust Bonds Aren't Policy — drafted (Faber)
  ...

What's blocked:
  · Blog route pending koad merge
  · fourty4 API auth (#44)
  ...

Week 7 preview:
  · Chiron gestation
  · First 5 sponsors target
  ...

────────────────────────────────────────────────────
[← Week 5]                              [Archive ↓]
```

- Font: Inter for body, JetBrains Mono for entity names, issue numbers, commit hashes
- No decorative header images
- "Archive" expands an accordion showing past digest links — lightweight, no infinite scroll

## Section 3: The Ring — Member Directory

**What it is:** Who's in the ring. This is a trust network, not a follower count. Showing ring members to each other is itself the value.

**Sovereignty constraint:** No third-party avatar services. No Gravatar, no GitHub CDN avatars via `avatars.githubusercontent.com` (these phone home and create third-party requests). Members are listed by name and namespace only.

**What to show per member:**

```
┌──────────────────────────────────────────────────────────┐
│  koad                                                    │
│  kingofalldata.com · koad.sh               Ring since March 2026  │
│  Creator. koad:io framework author.                      │
└──────────────────────────────────────────────────────────┘
```

Fields:
- Display name (human name or handle — their choice)
- Namespace (e.g. `username.koad.io` when live, else link to GitHub profile)
- Optional one-line bio (member-submitted, stored in member record)
- "Ring since [month year]" — date precision is month only (no exact day — less surveillance-adjacent)
- Tier is **not** displayed publicly within the ring. Tiers are between the member and koad. Peers don't need to know who paid more.

**Entity + human name pairs:**
Some members will have both a human name and an entity name. Where both exist, show them together:

```
koad (Jason Zvaniga)
koad:io author · thinker, flowbie, fourty4
```

The entity and human are the same person in different registers. Show both without hierarchy.

**Ordering:** Chronological by ring join date, oldest first. No engagement ranking. No "top member" sorting. Seniority of commitment is the only meaningful order here.

**Count:** Show total ring members in the section header. Just the number — no commentary.

```
RING  ·  4 members
```

## Section 4: Unreleased Entity Access

**What it is:** Entities in development, not yet public, accessible to ring members before general availability.

**Access mechanism: Git clone from private GitHub repo.** No custom download infrastructure. No ZIP files. The entity is a git repo — members clone it the same way anyone else would once it's public. They just get the private URL earlier.

**Why this is the right mechanism:**
- Consistent with sovereignty principle — the entity lives on the member's disk immediately
- No custom access management layer to build
- The "early access" is literally structural: the repo is private, ring members are GitHub collaborators, they can clone and pull updates

**Implementation:**
- Juno adds Level 1+ sponsors as collaborators on the private entity repos via GitHub API
- Members see the list here with clone instructions
- When the entity goes public, the clone URL doesn't change — they already have it

**UX:**

```
ACCESS  ·  3 entities in early access

──────────────────────────────────────────────────────
veritas         Quality guardian          [pre-release]
──────────────────────────────────────────────────────
Fact-checking and QA entity. Not yet publicly available.

  git clone git@github.com:koad/veritas.git ~/.veritas

──────────────────────────────────────────────────────
mercury         Communications            [pre-release]
──────────────────────────────────────────────────────
Comms and social entity. Not yet publicly available.

  git clone git@github.com:koad/mercury.git ~/.mercury

──────────────────────────────────────────────────────
sibyl           Research & analysis       [pre-release]
──────────────────────────────────────────────────────
Research entity. Not yet publicly available.

  git clone git@github.com:koad/sibyl.git ~/.sibyl
```

- Clone command: copy-on-click (same amber flash pattern from directory brief)
- Status badge: `[pre-release]` in amber, `[public]` in muted green (auto-updates when entity goes public and repo becomes public)
- No instructions beyond the git clone line. If someone is in the Insiders ring, they know what to do with a git repo.

**Level gating:** All tiers see this section. The entities are the same across tiers — early access is the benefit, not tier-differentiated content.

## Section 5: The Channel

**What it is:** Where ring members talk to each other and to Juno directly.

**Where it lives: Keybase.** The Insiders group on Keybase is the designated channel per `SPONSORS.md`. Keybase is end-to-end encrypted, sovereign-adjacent (keys matter there too), and does not require a phone number or a real name — consistent with the ethos of this project.

The Discord mention in `SPONSORS.md` is listed as a fallback. At this stage, lead with Keybase. If demand pushes toward Discord, that's a future decision — don't design two channels simultaneously.

**Joining flow:**

```
CHANNEL

The Insiders is on Keybase.

1. Install Keybase: keybase.io/download
2. Create an account (any username)
3. Send koad your Keybase username:
   · In a reply to your GitHub Sponsors welcome message, or
   · Via GitHub issue on koad/juno with label "insiders-invite"

koad adds you manually. You'll receive a Keybase team invite.
You're in.
```

The manual add step is intentional at this stage. There are few enough members that automation isn't needed and the manual step is itself a signal that this is real and personal, not SaaS onboarding.

**Join CTA:**

```
┌────────────────────────────────────────────────────────┐
│                                                        │
│  Not in the Keybase channel yet?                       │
│                                                        │
│  File a GitHub issue →  koad/juno · label: insiders-invite  │
│                                                        │
│  koad will add you within 48 hours.                    │
│                                                        │
└────────────────────────────────────────────────────────┘
```

- Link to `github.com/koad/juno/issues/new` pre-filled with the label (via URL parameter)
- "within 48 hours" — honest and realistic, not instant-gratification
- No "Join Discord" button. One channel. One instruction.

## Visual Tone

This surface is **not** the Alice experience (warm, guiding, soft). It is also not the homepage (public, evangelizing). It sits closer to:

- A private wiki that happens to be well-designed
- Terminal output that someone has taken care over
- A letter from a person, not a product

**Typography:** Inter for prose, JetBrains Mono for all identifiers (namespaces, entity names, commands, commit hashes). No display fonts. No oversized hero text.

**Color temperature:** Dark theme, same token set as the rest of the site. Accent color (`--accent`, violet) reserved for tier badges and interactive elements. Section headers in muted uppercase tracking. No decorative gradients.

**Whitespace:** Generous. This is a document, not a grid of cards. The breathing room communicates that what's here is worth reading.

**Border style:** Single-pixel `--border` on card-like blocks. No shadows, no border-radius inflation. Flat, solid, honest.

## Implementation Notes (Meteor/Blaze)

- Auth: GitHub OAuth via Meteor `accounts-github`. Scope: `read:user`, `read:org` (for sponsor tier check via GitHub API)
- Digest fetch: `Meteor.call('getLatestDigest')` server method; server holds GitHub API token for private repo access; client never sees the token
- Member list: `Members` collection, server-side populated from GitHub Sponsors API on a scheduled job (hourly). Only exposes fields the member has consented to show: name, namespace, bio, join date (month+year only)
- Early access entity list: `EarlyAccessEntities` collection, manually managed by Juno. Each record: entity name, description, repo slug, public date (null if pre-release)
- Clone URL visibility: only rendered for authenticated Level 1+ members. Server-side check, not just CSS hide.
- Keybase join issue link: static `href` with pre-filled title and label via GitHub issue URL params
